# Codex Desktop / codex-shim 升级后操作手册

本手册覆盖两类常见升级，分别给出完整操作流程：

| 场景 | 触发条件 | 影响范围 |
|---|---|---|
| **A** | Codex Desktop 客户端升级 | `/Applications/Codex.app` 二进制被更新；之前打的 picker 补丁被清掉，可能 needle 也需要重写 |
| **B** | codex-shim 上游 `0xSero/codex-shim` 有新提交 | 仓库 Python 代码可能落后于官方；可能错过新功能或修复 |
| **C** | A、B 同时发生 | 综合处理（推荐"先 git 再 patch"顺序） |

**重要心智模型**：A 和 B 是**独立**的两件事，没有强依赖。

- A 必做 `patch-app`，跟上游 git 无关
- B 不一定要 `patch-app`，要看上游改了什么文件
- 遇到 A 时**顺手 fetch 上游**是好习惯，因为官方可能正好刚发了适配新 Desktop 的 needle 修复，让你少改东西

> 所有命令默认在仓库根目录 `/Users/maerun/Documents/Third-Party/codex-shim` 下执行。

## 场景 A：Codex Desktop 客户端升级

### A-1. 升级前准备

**(1) 把你 fork 当前可用代码提交并推走，留作回滚锚点**

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
git status
# 如果有未提交改动：
git add -A
git commit -m "snapshot: working state before Codex Desktop upgrade"
git push origin feature/smart-vision-routing
```

**(2) 把 Codex Desktop 还原到官方原版**

让升级器从干净起点工作，避免补丁残留 + 升级互相干扰。

```bash
sudo "$(which codex-shim)" restore-app
```

**(3) 记录当前 Codex Desktop 版本号（便于对照）**

```bash
plutil -p /Applications/Codex.app/Contents/Info.plist | grep -i version
```

升级前版本：________

**(4) 顺手 fetch 上游，看官方有没有新修复**

```bash
git fetch upstream
git log --oneline HEAD..upstream/main
```

如果输出非空，建议跳到**场景 C** 走综合流程；输出为空再回到这里继续。

### A-2. 执行升级

走 Codex Desktop 自己的更新弹窗，或者手动下载 dmg 替换。完成后：

```bash
plutil -p /Applications/Codex.app/Contents/Info.plist | grep -i version
```

升级后版本：________

### A-3. 升级后修复

**(1) 检查并恢复 Codex provider 配置**

```bash
grep -A 5 codex_shim ~/.codex/config.toml
```

- 看到 `[model_providers.codex_shim]` 段 → 配置完好，跳到下一步
- 看不到 → 升级器把配置洗掉了，重新写入：

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
codex-shim enable
```

**(2) 直接尝试打补丁**

```bash
sudo "$(which codex-shim)" patch-app
```

输出会落入下面三种情况：

| 输出 | 含义 | 跳到 |
|---|---|---|
| 两条 `Patched ...` 或 `already applied` | 正则覆盖到了新版 bundle | A-4 |
| `Could not find the expected model picker allowlist filter ...` | picker 代码被重构 | 第四节侦察 |
| `Could not find the expected shim-mode sidebar provider filter ...` | sidebar shape 被调整 | 第五节侦察 |

### A-4. 重启 Codex Desktop 验证

```bash
osascript -e 'tell application "Codex" to quit' 2>/dev/null
killall Codex 2>/dev/null
sleep 1
open /Applications/Codex.app
```

打开后：

- 模型选择器里出现你 `~/.codex-shim/models.json` 配置的全部自定义模型 → ✅ 升级修复完成
- 仍然只显示官方默认那几个 → 跳到第四节侦察

### A-5. 终态验证

```bash
codex-shim status
codex-shim list
tail -n 30 .codex-shim/shim.log
```

CLI 跑一次实际请求验证路由正确：

```bash
codex-shim codex -- "用一句话告诉我现在是哪个模型在回答"
```

## 场景 B：codex-shim 上游有新提交

### B-1. 先看上游改了什么

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
git fetch upstream
git log --oneline HEAD..upstream/main           # 看上游领先了哪些 commit
git diff HEAD upstream/main -- codex_shim/      # 看 Python 代码具体改动
```

依据 diff 评估后续：

| 上游改了什么 | 后续动作 |
|---|---|
| `codex_shim/cli.py` 里 needle / `_patch_codex_desktop_bundles` | 合并后需要跑 `restore-app` + `patch-app` |
| `codex_shim/server.py` / `catalog.py` / `settings.py` / 路由逻辑 | 合并后需要 `codex-shim restart` 让守护进程加载新代码 |
| `tests/` / `README.md` / `docs/` / `pyproject.toml`（只改 dev 依赖） | 合并即可，**不需要**任何运行时动作 |
| `pyproject.toml` 的 runtime dependencies | 合并后重跑 `pip install --user -e ".[dev]" --break-system-packages` |

### B-2. 合并上游到你的 main，再变基你的开发分支

```bash
# 把上游同步到你 fork 的 main
git checkout main
git merge upstream/main
git push origin main

# 把你的开发分支变基到新 main 上
git checkout feature/smart-vision-routing
git rebase main
```

> rebase 过程中如果遇到冲突，按编辑器提示解决 → `git add <冲突文件>` → `git rebase --continue`。

变基完成后强制推到你 fork（因为 rebase 改写了 commit hash）：

```bash
git push origin feature/smart-vision-routing --force-with-lease
```

### B-3. 跑测试套件确认合并没崩

```bash
python3 -m pytest -x -q
```

失败的话先解决再继续。

### B-4. 根据 B-1 的评估执行运行时动作

**(a) 只需要重启 shim 守护进程**

```bash
codex-shim restart
codex-shim status
```

**(b) 需要重新打 Desktop 补丁**

```bash
sudo "$(which codex-shim)" restore-app
sudo "$(which codex-shim)" patch-app
osascript -e 'tell application "Codex" to quit' 2>/dev/null
killall Codex 2>/dev/null
sleep 1
open /Applications/Codex.app
```

**(c) 需要重装依赖**

```bash
python3 -m pip install --user -e ".[dev]" --break-system-packages
codex-shim restart
```

## 场景 C：两者同时发生

合理顺序是**先 git 再 patch-app** —— 上游可能正好刚发了适配新 Desktop 的修复，先合上来能少自己改 needle。

```bash
# 1. 同步上游（场景 B 流程压缩版）
cd /Users/maerun/Documents/Third-Party/codex-shim
git fetch upstream
git log --oneline HEAD..upstream/main           # 确认有新提交
git checkout main && git merge upstream/main && git push origin main
git checkout feature/smart-vision-routing
git rebase main
git push origin feature/smart-vision-routing --force-with-lease

# 2. 跑测试确认合并没崩
python3 -m pytest -x -q

# 3. 升级 Codex Desktop（走官方更新弹窗 / 手动 dmg）

# 4. 恢复 config + 打补丁（场景 A 流程压缩版）
grep -A 5 codex_shim ~/.codex/config.toml || codex-shim enable
sudo "$(which codex-shim)" restore-app
sudo "$(which codex-shim)" patch-app

# 5. 重启 Desktop 验证
osascript -e 'tell application "Codex" to quit' 2>/dev/null
killall Codex 2>/dev/null
sleep 1
open /Applications/Codex.app
```

如果 patch-app 在第 4 步失败，跳到第四 / 五节侦察流程。

## 四、Picker 补丁找不到时的侦察流程（场景 A / C 失败时进入）

### 4.1 解包当前 Codex Desktop 的 app.asar

```bash
cd /tmp && rm -rf codex-asar-inspect && mkdir codex-asar-inspect && cd codex-asar-inspect
npx --yes @electron/asar extract /Applications/Codex.app/Contents/Resources/app.asar extracted
```

### 4.2 定位 useHiddenModels 周围上下文

```bash
grep -RIno '.\{40\}useHiddenModels.\{60\}' extracted/webview/assets/ 2>/dev/null | head -10
```

### 4.3 找出 amazonBedrock 出现在哪些 bundle 里

```bash
grep -RIln 'amazonBedrock' extracted/webview/assets/ 2>/dev/null
```

> 过滤逻辑搬到了哪几个文件，这一步直接给出答案。

### 4.4 把所有候选文件美化导出到仓库内

```bash
TARGET_DIR=/Users/maerun/Documents/Third-Party/codex-shim/.tmp-bundle-inspect
rm -rf "$TARGET_DIR" && mkdir -p "$TARGET_DIR"

for f in $(grep -RIln 'amazonBedrock' extracted/webview/assets/*.js 2>/dev/null); do
  name=$(basename "$f")
  npx --yes prettier "$f" > "$TARGET_DIR/${name}.pretty.js"
done

ls -la "$TARGET_DIR"
```

### 4.5 把侦察结果交给 Claude / 自行分析

带着以下信息回到对话窗口请 Claude 帮忙更新 needle：

- 4.2 的 grep 输出
- 4.3 的文件列表
- 4.4 列出的所有文件名（特别留意几 KB 的小文件，过滤函数通常被抽进小文件）

需要修改的源码位置：`codex_shim/cli.py` 顶部 `MODEL_PICKER_NEEDLE` / `MODEL_PICKER_REPLACEMENT` / `MODEL_PICKER_APPLIED` 三个常量，以及 `_patch_codex_desktop_bundles` 里的 glob 列表。

改完跑测试再实战：

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
python3 -m pytest -x -q
sudo "$(which codex-shim)" restore-app
sudo "$(which codex-shim)" patch-app
```

## 五、Sidebar 补丁找不到时的侦察流程

```bash
cd /tmp/codex-asar-inspect
grep -RIno 'listRecentThreads([^)]*){[^}]*}' extracted/webview/assets/ 2>/dev/null | head -3
```

把命中行贴出来，对照 `codex_shim/cli.py` 里 `SIDEBAR_RECENT_THREADS_NEEDLE` / `APPLIED` 找差异，通常只需替换几个字符（变量名、`null` ↔ `[]`、`sourceKinds:xx` 等）。

如果新版本已经天然采用 `modelProviders:[]`（官方主动合并了你的修复），sidebar 补丁应该自动识别为 `already applied`，**不需要改动代码**。

## 六、补丁有效后，把修复回 git 并考虑上游 PR

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim

# 跑测试
python3 -m pytest -x -q

# 清理临时侦察目录（已 gitignore 但仍可手动清掉）
rm -rf .tmp-bundle-inspect

# 提交
git add codex_shim/cli.py tests/test_settings_catalog.py
git commit -m "fix: adapt picker needle to Codex Desktop <版本号> bundle layout"
git push origin feature/smart-vision-routing
```

这种"追新 needle"的修复非常适合回 PR 给上游：

```bash
git checkout main
git pull upstream main
git checkout -b fix/picker-needle-<版本号>
git cherry-pick <上一步 commit 的 hash>
git push origin fix/picker-needle-<版本号>
# 然后到 GitHub 上对 0xSero/codex-shim:main 开 PR
```

## 七、不打补丁的应急路径

如果新版 Codex Desktop 一时半会儿适配不出来，又急着用 BYOK 模型，可以**完全绕开 Desktop UI**：

```bash
# 浏览器选择器，不依赖任何 ASAR 补丁
open http://127.0.0.1:8765/picker

# 命令行直接跑
codex-shim codex -- "你的提示词"

# 想固定默认模型给后续 CLI 调用
codex-shim model use <slug>
```

这两条路径只用到 shim 后端（HTTP 127.0.0.1:8765）和 `~/.codex/config.toml` 受管理块，跟 ASAR 补丁完全无关，Codex Desktop 怎么改都不影响。

## 八、紧急回滚

### 8.1 回滚 Codex Desktop 到升级前

```bash
ls /Applications/ | grep Codex.app.unpatched
sudo rm -rf /Applications/Codex.app
sudo mv "/Applications/Codex.app.unpatched-<最近的时间戳>" /Applications/Codex.app
sudo "$(which codex-shim)" patch-app
```

注意：`patch-app` 在第一次执行前会写入 `.unpatched-*` 备份。如果你从未跑过 `patch-app`，没有这层备份，必须靠 Codex Desktop 升级器自身保留的旧版本或从外部 dmg 重装。

### 8.2 回滚 codex-shim 代码到合并上游前

如果 B-2 合并上游后引入了新问题：

```bash
# 看刚才合并/变基前的 reflog
git reflog | head -20

# 回到合并前那个状态（找到形如 'HEAD@{2}: rebase finished' 之前的那条）
git reset --hard HEAD@{N}
git push origin feature/smart-vision-routing --force-with-lease
```

## 九、常见坑速查

| 现象 | 排查方向 |
|---|---|
| `Operation not permitted` 写不进 `/Applications/Codex.app/...` | 系统设置 → 隐私与安全性 → App 管理，给当前终端开权限，重启终端 |
| `Could not find the expected model picker allowlist filter` | 新版重构了 picker → 走第四节侦察 |
| `Could not find the expected shim-mode sidebar provider filter` | sidebar shape 改了 → 走第五节侦察 |
| `404` upstream 错误，日志里 URL 带 `/v1` 重复段 | `_join_url` 的版本段识别可能又遇到新模式（比如 `/v5`），检查 `codex_shim/server.py` 里 `_join_url` |
| `[err] upstream ... returned 401/403` | API key 失效或权限不足，与升级无关，检查 `~/.codex-shim/models.json` |
| 选择器出现自定义模型但点了不工作 | 看 `tail -f .codex-shim/shim.log`，多半是上游 API endpoint 变了 |
| 升级后 shim 守护进程僵死 | `codex-shim restart`；仍不行删除 `.codex-shim/shim.pid` 后再 `codex-shim start` |
| rebase 时反复冲突 | 用 `git rebase --abort` 退出，改用 `git merge upstream/main`（避免 force push） |
| `git push` 报 non-fast-forward | rebase 后必须用 `--force-with-lease` 而不是普通 push |

## 附录 B：场景 A 一键修复脚本

把场景 A 的顺利路径封装成脚本，每次 Desktop 升级完跑一次：

```bash
cat > ~/codex-shim-refresh.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

REPO=/Users/maerun/Documents/Third-Party/codex-shim
cd "$REPO"

echo "==> Checking for upstream updates"
git fetch upstream
if [[ -n "$(git log --oneline HEAD..upstream/main 2>/dev/null)" ]]; then
  echo "Upstream has new commits — consider scenario C in 升级后操作手册.md before continuing."
  echo "Run anyway? [y/N]"
  read -r ans
  [[ "$ans" =~ ^[Yy] ]] || exit 0
fi

echo "==> Restoring Codex provider config if missing"
if ! grep -q codex_shim ~/.codex/config.toml 2>/dev/null; then
  codex-shim enable
fi

echo "==> Restoring app.asar to a clean state"
sudo "$(which codex-shim)" restore-app || true

echo "==> Restarting shim daemon"
codex-shim restart

echo "==> Applying picker patch"
if ! sudo "$(which codex-shim)" patch-app; then
  echo
  echo "patch-app failed — likely an upstream layout change."
  echo "Follow the manual inspection flow in 升级后操作手册.md (section 四 / 五)."
  exit 1
fi

echo "==> Restarting Codex Desktop"
osascript -e 'tell application "Codex" to quit' 2>/dev/null || true
killall Codex 2>/dev/null || true
sleep 1
open /Applications/Codex.app

echo "==> Done. Open the model picker to confirm your custom models show up."
EOF
chmod +x ~/codex-shim-refresh.sh
```

之后每次 Desktop 升级完跑：

```bash
~/codex-shim-refresh.sh
```

脚本会自动检查上游，提示你是否需要先走场景 C；patch-app 失败也会直接退出并提示去第四 / 五节侦察。

## 附录 C：场景 B 一键拉取脚本

```bash
cat > ~/codex-shim-sync-upstream.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

REPO=/Users/maerun/Documents/Third-Party/codex-shim
BRANCH=feature/smart-vision-routing
cd "$REPO"

echo "==> Fetching upstream"
git fetch upstream

NEW=$(git log --oneline HEAD..upstream/main 2>/dev/null)
if [[ -z "$NEW" ]]; then
  echo "No upstream updates. Bye."
  exit 0
fi

echo "==> Upstream commits ahead:"
echo "$NEW"
echo

echo "==> Merging upstream/main into your fork's main"
git checkout main
git merge upstream/main
git push origin main

echo "==> Rebasing $BRANCH onto main"
git checkout "$BRANCH"
git rebase main
git push origin "$BRANCH" --force-with-lease

echo "==> Running tests"
python3 -m pytest -x -q

echo
echo "==> Done. Review what changed:"
echo "    git diff HEAD~1 HEAD -- codex_shim/"
echo
echo "If cli.py picker/sidebar logic changed, also run:"
echo "    sudo \"\$(which codex-shim)\" restore-app && sudo \"\$(which codex-shim)\" patch-app"
echo "Otherwise just:"
echo "    codex-shim restart"
EOF
chmod +x ~/codex-shim-sync-upstream.sh
```

每次想同步上游时跑：

```bash
~/codex-shim-sync-upstream.sh
```

跑完它会提示你下一步是否需要 patch-app 或只需 restart。
