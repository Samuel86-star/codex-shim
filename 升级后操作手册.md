# Codex Desktop 升级后操作手册

每次 Codex Desktop 自动更新或手动升级之后，都会做以下三件事让 codex-shim 失效：

1. 覆盖 `/Applications/Codex.app/Contents/Resources/app.asar`，之前打的 picker 补丁消失，allowlist 重新生效
2. 可能改写或覆盖 `~/.codex/config.toml` 里的受管理块，shim provider 配置丢失
3. 重新混淆 JS bundle 的变量名甚至重构整个过滤逻辑，原 needle 可能匹配不上

本文档给出一套**升级前 → 升级中 → 升级后**的完整流程，按顺序走就行。

> 所有命令默认在仓库根目录 `/Users/maerun/Documents/Third-Party/codex-shim` 执行。

---

## 一、升级前准备（重要）

### 1. 提交并推送当前可用版本到你的 fork

万一升级后新版本暂时打不出补丁，你随时可以回滚 Desktop 到当前可用版本，已修复的代码不会丢。

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
git status
# 如果有未提交改动：
git add -A
git commit -m "snapshot: working state before Codex Desktop upgrade"
git push origin feature/smart-vision-routing
```

### 2. 把 Codex Desktop 还原到官方原版

让升级器从干净起点工作，避免补丁 + 升级互相干扰。

```bash
sudo "$(which codex-shim)" restore-app
```

### 3. 记录当前 Codex Desktop 版本号

升级后对照用。

```bash
plutil -p /Applications/Codex.app/Contents/Info.plist | grep -i version
```

把输出记到这里 → **升级前版本：________**

---

## 二、执行升级

走 Codex Desktop 自己的更新弹窗，或者手动下载 dmg 替换。

升级完成后再记录一次新版本号 → **升级后版本：________**

---

## 三、升级后修复（按顺序执行）

### 步骤 1：检查并恢复 Codex provider 配置

```bash
grep -A 5 codex_shim ~/.codex/config.toml
```

**如果看到 `[model_providers.codex_shim]` 段** → 配置完好，跳到步骤 2。

**如果看不到** → 升级器把配置洗掉了，重新写入：

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
codex-shim enable
```

### 步骤 2：直接尝试打补丁

```bash
sudo "$(which codex-shim)" patch-app
```

输出会落入下面三种情况之一，分别对应不同动作：

| 输出 | 含义 | 跳到 |
|---|---|---|
| 两条 `Patched ...` 或 `already applied` | 正则覆盖到了新版 bundle | 步骤 3 |
| `Could not find the expected model picker allowlist filter ...` | picker 代码被重构，需要找新补丁点 | 步骤 4 |
| `Could not find the expected shim-mode sidebar provider filter ...` | sidebar shape 被调整 | 步骤 5 |

### 步骤 3：重启 Codex Desktop 验证

```bash
osascript -e 'tell application "Codex" to quit' 2>/dev/null
killall Codex 2>/dev/null
sleep 1
open /Applications/Codex.app
```

打开后：

- 模型选择器里出现你 `~/.codex-shim/models.json` 配置的全部自定义模型 → ✅ 升级修复完成
- 仍然只显示官方默认那几个 → 跳到步骤 4

## 四、Picker 补丁找不到时的侦察流程

### 1. 解包当前 Codex Desktop 的 app.asar

```bash
cd /tmp && rm -rf codex-asar-inspect && mkdir codex-asar-inspect && cd codex-asar-inspect
npx --yes @electron/asar extract /Applications/Codex.app/Contents/Resources/app.asar extracted
```

### 2. 定位 useHiddenModels 周围上下文

```bash
grep -RIno '.\{40\}useHiddenModels.\{60\}' extracted/webview/assets/ 2>/dev/null | head -10
```

把所有命中行复制下来。

### 3. 找出 amazonBedrock 出现在哪些 bundle 里

```bash
grep -RIln 'amazonBedrock' extracted/webview/assets/ 2>/dev/null
```

这一步告诉我们过滤逻辑搬到了哪几个文件。

### 4. 把所有候选文件美化导出到仓库内

```bash
TARGET_DIR=/Users/maerun/Documents/Third-Party/codex-shim/.tmp-bundle-inspect
rm -rf "$TARGET_DIR" && mkdir -p "$TARGET_DIR"

for f in $(grep -RIln 'amazonBedrock' extracted/webview/assets/*.js 2>/dev/null); do
  name=$(basename "$f")
  npx --yes prettier "$f" > "$TARGET_DIR/${name}.pretty.js"
done

ls -la "$TARGET_DIR"
```

### 5. 把侦察结果交给 Claude / 自行分析

带着以下信息回到对话窗口请 Claude 帮忙更新 needle：

- 步骤 2 的 grep 输出
- 步骤 3 的文件列表
- 步骤 4 列出的所有文件名（特别留意几 KB 的小文件，过滤函数通常被抽进小文件）

需要修改的源码位置：`codex_shim/cli.py` 顶部 `MODEL_PICKER_NEEDLE` / `MODEL_PICKER_REPLACEMENT` / `MODEL_PICKER_APPLIED` 三个常量，以及 `_patch_codex_desktop_bundles` 里的 glob 列表。

改完跑测试再实战：

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim
python3 -m pytest -x -q
sudo "$(which codex-shim)" restore-app
sudo "$(which codex-shim)" patch-app
```

## 五、Sidebar 补丁找不到时的侦察流程

```bash
cd /tmp/codex-asar-inspect
grep -RIno 'listRecentThreads([^)]*){[^}]*}' extracted/webview/assets/ 2>/dev/null | head -3
```

把命中行贴出来，对照 `codex_shim/cli.py` 里 `SIDEBAR_RECENT_THREADS_NEEDLE` / `APPLIED` 找差异，通常只需替换几个字符（变量名、`null` ↔ `[]`、`sourceKinds:xx` 等）。

如果新版本已经天然采用 `modelProviders:[]`（官方主动合并了我们的修复），sidebar 补丁应该自动识别为 `already applied`，**不需要改动代码**。

---

## 六、补丁成功后的最终验证

### 1. 重启 Codex Desktop

```bash
osascript -e 'tell application "Codex" to quit' 2>/dev/null
killall Codex 2>/dev/null
sleep 1
open /Applications/Codex.app
```

### 2. 检查模型选择器

应该出现 `~/.codex-shim/models.json` 里配置的全部自定义模型。

### 3. 检查 shim 守护进程

```bash
codex-shim status
tail -n 30 .codex-shim/shim.log
```

### 4. 跑一次实际请求验证路由正确

CLI 方式：

```bash
codex-shim codex -- "用一句话告诉我现在是哪个模型在回答"
```

或浏览器打开 Web Picker：

```
http://127.0.0.1:8765/picker
```

切换模型，发一条消息，确认日志里 `[route]` 的 `slug` 和 `url` 正确，没有 404。

## 七、如果新补丁有效，提交修复

```bash
cd /Users/maerun/Documents/Third-Party/codex-shim

# 跑测试
python3 -m pytest -x -q

# 清理临时侦察目录（已被 gitignore 但仍可手动清掉）
rm -rf .tmp-bundle-inspect

# 提交
git add codex_shim/cli.py tests/test_settings_catalog.py
git commit -m "fix: adapt picker needle to Codex Desktop <版本号> bundle layout"
git push origin feature/smart-vision-routing
```

这种"上游升级后追新 needle"的修复非常适合回 PR 给上游 `0xSero/codex-shim`，可以让所有用户受益：

```bash
git checkout main
git pull upstream main
git checkout -b fix/picker-needle-<版本号>
git cherry-pick <刚才提交的 commit hash>
git push origin fix/picker-needle-<版本号>
# 然后到 GitHub 上开 PR 到 0xSero/codex-shim:main
```

## 八、不打补丁的应急路径

如果新版 Codex Desktop 一时半会儿适配不出来，又急着用 BYOK 模型，可以**完全绕开 Desktop UI**：

```bash
# 浏览器选择器，不依赖任何 ASAR 补丁
open http://127.0.0.1:8765/picker

# 或命令行直接跑
codex-shim codex -- "你的提示词"

# 想固定默认模型给后续 CLI 调用
codex-shim model use <slug>
```

这两条路径只用到 shim 后端（HTTP 127.0.0.1:8765）和 `~/.codex/config.toml` 受管理块，跟 ASAR 补丁完全无关，Codex Desktop 怎么改都不影响。

## 九、紧急回滚

如果新版 Desktop 即便不打补丁也彻底用不了（比如选择器渲染崩溃、provider 解析失败等），可以回滚到升级前那一份 `.unpatched-<时间戳>` 备份：

```bash
ls /Applications/ | grep Codex.app.unpatched
sudo rm -rf /Applications/Codex.app
sudo mv "/Applications/Codex.app.unpatched-<最近的时间戳>" /Applications/Codex.app

# 重新打回旧版本的补丁
sudo "$(which codex-shim)" patch-app
```

注意：`patch-app` 在升级前会写入备份，所以 `restore-app` 能恢复到 patch-app 之前的状态；但如果你之前没跑过 `patch-app`，没有这层备份，必须靠 Codex Desktop 升级器自身保留的旧版本或从外部 dmg 重装。

## 十、常见坑速查

| 现象 | 排查方向 |
|---|---|
| `Operation not permitted` 写不进 `/Applications/Codex.app/...` | 系统设置 → 隐私与安全性 → App 管理，给当前终端开权限，重启终端 |
| `Could not find the expected model picker allowlist filter` | 新版重构了 picker → 走第四节侦察流程 |
| `Could not find the expected shim-mode sidebar provider filter` | sidebar shape 改了 → 走第五节侦察流程 |
| `404` upstream 错误，日志里 URL 带 `/v1` 重复段 | `_join_url` 的版本段识别可能又遇到新模式（比如 `/v5`），检查 `codex_shim/server.py` 里 `_join_url` |
| `[err] upstream ... returned 401/403` | API key 失效或权限不足，与升级无关，检查 `~/.codex-shim/models.json` |
| 选择器出现自定义模型但点了不工作 | 看 `tail -f .codex-shim/shim.log`，多半是上游 API endpoint 变了 |
| 升级后 shim 守护进程僵死 | `codex-shim restart`；仍不行删除 `.codex-shim/shim.pid` 后再 `codex-shim start` |

## 附录：一键修复脚本

把日常顺利流程封装成脚本，每次升级后跑一次：

```bash
cat > ~/codex-shim-refresh.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

REPO=/Users/maerun/Documents/Third-Party/codex-shim
cd "$REPO"

echo "==> Restarting shim daemon"
codex-shim restart

echo "==> Restoring app.asar to a clean state"
sudo "$(which codex-shim)" restore-app || true

echo "==> Applying picker patch"
if ! sudo "$(which codex-shim)" patch-app; then
  echo
  echo "patch-app failed — likely an upstream layout change."
  echo "Follow the manual inspection flow in 升级后操作手册.md (section 四 / 五)."
  exit 1
fi

echo "==> Restarting Codex Desktop"
osascript -e 'tell application "Codex" to quit' 2>/dev/null || true
killall Codex 2>/dev/null || true
sleep 1
open /Applications/Codex.app

echo "==> Done. Open the model picker to confirm your custom models show up."
EOF
chmod +x ~/codex-shim-refresh.sh
```

之后每次升级完跑：

```bash
~/codex-shim-refresh.sh
```

脚本里如果 `patch-app` 失败会直接退出并提示你回到本文档第四 / 五节走侦察流程。
